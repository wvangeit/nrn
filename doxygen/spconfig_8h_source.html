46 /*
47  * OPTIONS
48  *
49  * These are compiler options. Set each option to one to compile that
50  * section of the code. If a feature is not desired, set the macro
51  * to NO. Recommendations are given in brackets, [ignore them].
52  *
53  * >>> Option descriptions:
54  * Arithmetic Precision
55  * The precision of the arithmetic used by Sparse can be set by
56  * changing changing the spREAL macro. This macro is
57  * contained in the file spMatrix.h. It is strongly suggested to
58  * used double precision with circuit simulators. Note that
59  * because C always performs arithmetic operations in double
60  * precision, the only benefit to using single precision is that
61  * less storage is required. There is often a noticeable speed
62  * penalty when using single precision. Sparse internally refers
63  * to a spREAL as a RealNumber.
64  * REAL
65  * This specifies that the routines are expected to handle real
66  * systems of equations. The routines can be compiled to handle
67  * both real and complex systems at the same time, but there is a
68  * slight speed and memory advantage if the routines are complied
69  * to handle only real systems of equations.
70  * spCOMPLEX
71  * This specifies that the routines will be complied to handle
72  * complex systems of equations.
73  * EXPANDABLE
74  * Setting this compiler flag true (1) makes the matrix
75  * expandable before it has been factored. If the matrix is
76  * expandable, then if an element is added that would be
77  * considered out of bounds in the current matrix, the size of
78  * the matrix is increased to hold that element. As a result,
79  * the size of the matrix need not be known before the matrix is
80  * built. The matrix can be allocated with size zero and
81  * expanded.
82  * TRANSLATE
83  * This option allows the set of external row and column numbers
84  * to be non-packed. In other words, the row and column numbers
85  * do not have to be contiguous. The priced paid for this
86  * flexibility is that when TRANSLATE is set true, the time
87  * required to initially build the matrix will be greater because
88  * the external row and column number must be translated into
89  * internal equivalents. This translation brings about other
90  * benefits though. First, the spGetElement() and
91  * spGetAdmittance() routines may be used after the matrix has
92  * been factored. Further, elements, and even rows and columns,
93  * may be added to the matrix, and row and columns may be deleted
94  * from the matrix, after it has been factored. Note that when
95  * the set of row and column number is not a packed set, neither
96  * are the RHS and Solution vectors. Thus the size of these
97  * vectors must be at least as large as the external size, which
98  * is the value of the largest given row or column numbers.
99  * INITIALIZE
100  * Causes the spInitialize(), spGetInitInfo(), and
101  * spInstallInitInfo() routines to be compiled. These routines
102  * allow the user to store and read one pointer in each nonzero
103  * element in the matrix. spInitialize() then calls a user
104  * specified function for each structural nonzero in the matrix,
105  * and includes this pointer as well as the external row and
106  * column numbers as arguments. This allows the user to write
107  * custom matrix initialization routines.
108  * DIAGONAL_PIVOTING
109  * Many matrices, and in particular node- and modified-node
110  * admittance matrices, tend to be nearly symmetric and nearly
111  * diagonally dominant. For these matrices, it is a good idea to
112  * select pivots from the diagonal. With this option enabled,
113  * this is exactly what happens, though if no satisfactory pivot
114  * can be found on the diagonal, an off-diagonal pivot will be
115  * used. If this option is disabled, Sparse does not
116  * preferentially search the diagonal. Because of this, Sparse
117  * has a wider variety of pivot candidates available, and so
118  * presumably fewer fill-ins will be created. However, the
119  * initial pivot selection process will take considerably longer.
120  * If working with node admittance matrices, or other matrices
121  * with a strong diagonal, it is probably best to use
122  * DIAGONAL_PIVOTING for two reasons. First, accuracy will be
123  * better because pivots will be chosen from the large diagonal
124  * elements, thus reducing the chance of growth. Second, a near
125  * optimal ordering will be chosen quickly. If the class of
126  * matrices you are working with does not have a strong diagonal,
127  * do not use DIAGONAL_PIVOTING, but consider using a larger
128  * threshold. When DIAGONAL_PIVOTING is turned off, the following
129  * options and constants are not used: MODIFIED_MARKOWITZ,
130  * MAX_MARKOWITZ_TIES, and TIES_MULTIPLIER.
131  * ARRAY_OFFSET
132  * This determines whether arrays start at an index of zero or one.
133  * This option is necessitated by the fact that standard C
134  * convention dictates that arrays begin with an index of zero but
135  * the standard mathematic convention states that arrays begin with
136  * an index of one. So if you prefer to start your arrays with
137  * zero, or your calling Sparse from FORTRAN, set ARRAY_OFFSET to
138  * NO or 0. Otherwise, set ARRAY_OFFSET to YES or 1. Note that if
139  * you use an offset of one, the arrays that you pass to Sparse
140  * must have an allocated length of one plus the size of the
141  * matrix. ARRAY_OFFSET must be either 0 or 1, no other offsets
142  * are valid.
143  * spSEPARATED_COMPLEX_VECTORS
144  * This specifies the format for complex vectors. If this is set
145  * false then a complex vector is made up of one double sized
146  * array of RealNumber's in which the real and imaginary numbers
147  * are placed in the alternately array in the array. In other
148  * words, the first entry would be Complex[1].Real, then comes
149  * Complex[1].Imag, then Complex[1].Real, etc. If
150  * spSEPARATED_COMPLEX_VECTORS is set true, then each complex
151  * vector is represented by two arrays of RealNumbers, one with
152  * the real terms, the other with the imaginary. [NO]
153  * MODIFIED_MARKOWITZ
154  * This specifies that the modified Markowitz method of pivot
155  * selection is to be used. The modified Markowitz method differs
156  * from standard Markowitz in two ways. First, under modified
157  * Markowitz, the search for a pivot can be terminated early if a
158  * adequate (in terms of sparsity) pivot candidate is found.
159  * Thus, when using modified Markowitz, the initial factorization
160  * can be faster, but at the expense of a suboptimal pivoting
161  * order that may slow subsequent factorizations. The second
162  * difference is in the way modified Markowitz breaks Markowitz
163  * ties. When two or more elements are pivot candidates and they
164  * all have the same Markowitz product, then the tie is broken by
165  * choosing the element that is best numerically. The numerically
166  * best element is the one with the largest ratio of its magnitude
167  * to the magnitude of the largest element in the same column,
168  * excluding itself. The modified Markowitz method results in
169  * marginally better accuracy. This option is most appropriate
170  * for use when working with very large matrices where the initial
171  * factor time represents an unacceptable burden. [NO]
172  * DELETE
173  * This specifies that the spDeleteRowAndCol() routine
174  * should be compiled. Note that for this routine to be
175  * compiled, both DELETE and TRANSLATE should be set true.
176  * STRIP
177  * This specifies that the spStripFills() routine should be compiled.
178  * MODIFIED_NODAL
179  * This specifies that the routine that preorders modified node
180  * admittance matrices should be compiled. This routine results
181  * in greater speed and accuracy if used with this type of
182  * matrix.
183  * QUAD_ELEMENT
184  * This specifies that the routines that allow four related
185  * elements to be entered into the matrix at once should be
186  * compiled. These elements are usually related to an
187  * admittance. The routines affected by QUAD_ELEMENT are the
188  * spGetAdmittance, spGetQuad and spGetOnes routines.
189  * TRANSPOSE
190  * This specifies that the routines that solve the matrix as if
191  * it was transposed should be compiled. These routines are
192  * useful when performing sensitivity analysis using the adjoint
193  * method.
194  * SCALING
195  * This specifies that the routine that performs scaling on the
196  * matrix should be complied. Scaling is not strongly
197  * supported. The routine to scale the matrix is provided, but
198  * no routines are provided to scale and descale the RHS and
199  * Solution vectors. It is suggested that if scaling is desired,
200  * it only be preformed when the pivot order is being chosen [in
201  * spOrderAndFactor()]. This is the only time scaling has
202  * an effect. The scaling may then either be removed from the
203  * solution by the user or the scaled factors may simply be
204  * thrown away. [NO]
205  * DOCUMENTATION
206  * This specifies that routines that are used to document the
207  * matrix, such as spPrint() and spFileMatrix(), should be
208  * compiled.
209  * DETERMINANT
210  * This specifies that the routine spDeterminant() should be complied.
211  * STABILITY
212  * This specifies that spLargestElement() and spRoundoff() should
213  * be compiled. These routines are used to check the stability (and
214  * hence the quality of the pivoting) of the factorization by
215  * computing a bound on the size of the element is the matrix E =
216  * A - LU. If this bound is very high after applying
217  * spOrderAndFactor(), then the pivot threshold should be raised.
218  * If the bound increases greatly after using spFactor(), then the
219  * matrix should probably be reordered.
220  * CONDITION
221  * This specifies that spCondition() and spNorm(), the code that
222  * computes a good estimate of the condition number of the matrix,
223  * should be compiled.
224  * PSEUDOCONDITION
225  * This specifies that spPseudoCondition(), the code that computes
226  * a crude and easily fooled indicator of ill-conditioning in the
227  * matrix, should be compiled.
228  * MULTIPLICATION
229  * This specifies that the routines to multiply the unfactored
230  * matrix by a vector should be compiled.
231  * FORTRAN
232  * This specifies that the FORTRAN interface routines should be
233  * compiled. When interfacing to FORTRAN programs, the ARRAY_OFFSET
234  * options should be set to NO.
235  * DEBUG
236  * This specifies that additional error checking will be compiled.
237  * The type of error checked are those that are common when the
238  * matrix routines are first integrated into a user's program. Once
239  * the routines have been integrated in and are running smoothly, this
240  * option should be turned off.
241  */

287 /*
288  * MATRIX CONSTANTS
289  *
290  * These constants are used throughout the sparse matrix routines. They
291  * should be set to suit the type of matrix being solved. Recommendations
292  * are given in brackets.
293  *
294  * Some terminology should be defined. The Markowitz row count is the number
295  * of non-zero elements in a row excluding the one being considered as pivot.
296  * There is one Markowitz row count for every row. The Markowitz column
297  * is defined similarly for columns. The Markowitz product for an element
298  * is the product of its row and column counts. It is a measure of how much
299  * work would be required on the next step of the factorization if that
300  * element were chosen to be pivot. A small Markowitz product is desirable.
301  *
302  * >>> Constants descriptions:
303  * DEFAULT_THRESHOLD
304  * The relative threshold used if the user enters an invalid
305  * threshold. Also the threshold used by spFactor() when
306  * calling spOrderAndFactor(). The default threshold should
307  * not be less than or equal to zero nor larger than one. [0.001]
308  * DIAG_PIVOTING_AS_DEFAULT
309  * This indicates whether spOrderAndFactor() should use diagonal
310  * pivoting as default. This issue only arises when
311  * spOrderAndFactor() is called from spFactor().
312  * SPACE_FOR_ELEMENTS
313  * This number multiplied by the size of the matrix equals the number
314  * of elements for which memory is initially allocated in
315  * spCreate(). [6]
316  * SPACE_FOR_FILL_INS
317  * This number multiplied by the size of the matrix equals the number
318  * of elements for which memory is initially allocated and specifically
319  * reserved for fill-ins in spCreate(). [4]
320  * ELEMENTS_PER_ALLOCATION
321  * The number of matrix elements requested from the malloc utility on
322  * each call to it. Setting this value greater than 1 reduces the
323  * amount of overhead spent in this system call. On a virtual memory
324  * machine, its good to allocate slightly less than a page worth of
325  * elements at a time (or some multiple thereof).
326  * [For the VAX, for real only use 41, otherwise use 31]
327  * MINIMUM_ALLOCATED_SIZE
328  * The minimum allocated size of a matrix. Note that this does not
329  * limit the minimum size of a matrix. This just prevents having to
330  * resize a matrix many times if the matrix is expandable, large and
331  * allocated with an estimated size of zero. This number should not
332  * be less than one.
333  * EXPANSION_FACTOR
334  * The amount the allocated size of the matrix is increased when it
335  * is expanded.
336  * MAX_MARKOWITZ_TIES
337  * This number is used for two slightly different things, both of which
338  * relate to the search for the best pivot. First, it is the maximum
339  * number of elements that are Markowitz tied that will be sifted
340  * through when trying to find the one that is numerically the best.
341  * Second, it creates an upper bound on how large a Markowitz product
342  * can be before it eliminates the possibility of early termination
343  * of the pivot search. In other words, if the product of the smallest
344  * Markowitz product yet found and TIES_MULTIPLIER is greater than
345  * MAX_MARKOWITZ_TIES, then no early termination takes place.
346  * Set MAX_MARKOWITZ_TIES to some small value if no early termination of
347  * the pivot search is desired. An array of RealNumbers is allocated
348  * of size MAX_MARKOWITZ_TIES so it must be positive and shouldn't
349  * be too large. Active when MODIFIED_MARKOWITZ is 1 (true). [100]
350  * TIES_MULTIPLIER
351  * Specifies the number of Markowitz ties that are allowed to occur
352  * before the search for the pivot is terminated early. Set to some
353  * large value if no early termination of the pivot search is desired.
354  * This number is multiplied times the Markowitz product to determine
355  * how many ties are required for early termination. This means that
356  * more elements will be searched before early termination if a large
357  * number of fill-ins could be created by accepting what is currently
358  * considered the best choice for the pivot. Active when
359  * MODIFIED_MARKOWITZ is 1 (true). Setting this number to zero
360  * effectively eliminates all pivoting, which should be avoided.
361  * This number must be positive. TIES_MULTIPLIER is also used when
362  * diagonal pivoting breaks down. [5]
363  * DEFAULT_PARTITION
364  * Which partition mode is used by spPartition() as default.
365  * Possibilities include
366  * spDIRECT_PARTITION -- each row used direct addressing, best for
367  * a few relatively dense matrices.
368  * spINDIRECT_PARTITION -- each row used indirect addressing, best
369  * for a few very sparse matrices.
370  * spAUTO_PARTITION -- direct or indirect addressing is chosen on
371  * a row-by-row basis, carries a large overhead, but speeds up
372  * both dense and sparse matrices, best if there is a large
373  * number of matrices that can use the same ordering.
374  */
375
