76 (2002-04-06)
77 One reason for going into the following in depth is to allow
78 comparison of the fixed step implicit method
79 with the implementation of the corresponding
80 cvode (vext not allowed) and daspk (vext allowed) versions.
81 
82 In the fixed step method the result of a setup is a linear matrix
83 equation of the form
84  G*(vinew,vxnew) = i(vm,vx)
85 which is solved and followed by an update step
86  vx=vxnew
87  vm = vinew - vxnew
88 
89 When cvode was added, as much as possible of the existing setup was re-used
90 through the addition of flags. Cvode solves a system of the form
91  dy/dt = f(y)
92 and this is not obtainable in the presence of extracellular or LinearMechanism
93 instances. And even in the cases it can handle, it
94 required a good deal of manipulation to remove the zero area compartments
95 from the state vector and compute the algebraic equations separately.
96 In fact, as of 2002-04-06, the CONSERVE statement for kinetic schemes
97 in model descriptions is still ignored.
98 At any rate, the issues involved are
99 1) How to compute f(y). Any algebraic equations are solved inside f(y)
100 so that the algebraic states are consistent with y.
101 2) How to setup the jacobian (I - gamma*df/dy)*y = b
102 
103 When daspk was added which solves the form
104  0 = f(y, y')
105 having one implementation serving the setup for
106 three very distinct methods became too cumbersome. Therefore setup_tree_matrix
107 was copied twice and modified separately so that each method
108 had its own setup code derived from this file.
109 
110 The issues that have
111 arisen with daspk are
112 1) How to actually implement a residual function res = f(y, y')
113 2) How to actually implement a preconditioner setup of the form
114  (df/dy + cj*df/dy')*y = b
115 3) how to initialize so that 0 = f(y(0), y'(0))
116 when it is not possible to know which are the algebraic and which
117 are the differential equations. Also, it is often the case that
118 a linear combination
119 of apparently differential equations yields an algebraic equation and we
120 don't know the linearly dependent subsets.
121 DASPK itself will only handle initialization if one can mark some states
122 as differential and the remaining states as algebraic.
123 
124 With that, here's what is going on in this file with respect to setup of
125  G*(vinew, vxnew) = i(vm,vx)
126 
127 The current balance equations are sum of outward currents = sum of inward currents
128 For an internal node
129  cm*dvm/dt + i(vm) = is(vi) + ai_j*(vi_j - vi)
130 For an external node (one layer)
131  cx*dvx/dt + gx*(vx - ex) = cm*dvm/dt + i(vm) + ax_j*(vx_j - vx)
132 where vm = vi - vx
133 and every term has the dimensions mA/cm2
134 (the last terms in each equation represent the second derivative cable properties)
135 The chosen convention that determines the side of the terms and their sign
136 is that
137 positive membrane current is outward but electrode stimulus
138 currents are toward the node. We place the terms involving
139 currents leaving a node on the left, and currents entering a node
140 on the right. i.e.
141  |is
142  ia `.'
143  ---> <---
144  |im
145  ie `.'
146  ---> <---
147  |gx*vx
148  `.'
149  ---
150  \ /
151  -
152 
153 At the hoc level, v refers to vm and vext refers to vx.
154 It should also be realized that LinearMechanism instances overlay equations of
155 the form
156 c*dy/dt + g*y = b
157 onto the current balance equation set thus adding new states,
158 equations, and terms to the above equations. However,
159 note that when y refers to a voltage of a segment in those equations
160 it always means vi or vx and never vm.
161 
162 At this point it is up to us whether we setup and solve the form
163 G*(vm,vx) = i or G*(vi,vx) = i
164 since any particular choice is resolved back to (vm,vx) in the update step.
165 At present, particularly for the implicit fixed step method whose setup
166 implementation resides in this file, we've chosen G*(vi,vx) = i,
167 because there are fewer operations in setting up second derivatives and it
168 makes life easy for the LinearMechanism. i.e. on setup each element of the
169 linear mechanism gets added to only one element of the complete matrix equations
170 and on update we merely copy the
171 vi solution to the proper LinearMechanism states and update v=vi-vext
172 for the compartment membrane potentials.
173 
174 The implicit linear form for the compartment internal node is
175 cm/dt*(vmnew - vmold) + (di/dvm*(vmnew - vmold) + i(vmold))
176  = (dis/dvi*(vinew - viold) + is(viold)) + ai_j*(vinew_j - vinew)
177 and for the compartment external node is
178 cx/dt*(vxnew - vxold) - gx(vxnew - ex)
179  = cm/dt*(vmnew - vmold) + (di/dvm*(vmnew - vmold) + i(vmold))
180  + ax_j*(vxnew_j - vxnew) = 0
181 
182 and this forms the matrix equation row (for an internal node)
183  (cm/dt + di/dvm - dis/dvi + ai_j)*vinew
184  - ai_j*vinew_j
185  -(cm/dt + di/dvm)*vxnew
186  =
187  cm/dt*vmold + di/dvm*vmold - i(vmold) - dis/dvi*viold + is(viold)
188 
189 where old is present value of vm, vi, or vx and new will be found by solving
190 this matrix equation (which is of the form G*v = rhs)
191 The matrix equation row for the extracellular node is
192  (cx/dt + gx + ax_j + cm/dt + di/dvm)*vext
193  -ax_j*vxnew_j
194  -(cm/dt + di/dvm)*vinew
195  = cx/dt*vxold - (cm/dt*vmold + di/dv*vmold - i(vmold)) - gx*ex
196 
197 Node.v is always interpreted as membrane potential and it is important to
198 distinguish this from internal potential (Node.v + Node.extnode.v)
199 when the extracellular mechanism is present.
200 
201 In dealing with these current balance equations, all three methods
202 use the same mechanism calls to calculate di/dvm, i(vm), dis/dvi, is(vi)
203 and add them to the matrix equation.
204 Thus the passive channel, i=g*(v - e), is a transmembrane current
205 (convention is positive outward) and the mechanism adds g to the
206 diagonal element of the matrix and subtracts i - g*v from the right hand side.
207 (see nrn_cur of passive.cpp) It does NOT add anything to the extracellular
208 current balance row.
209 
210 An electrode current (convention positive current increases the internal potential)
211 such as a voltage clamp with small resistance, is=g*(vc - (v+vext))
212 (see nrn_cur of svclmp.cpp)
213 subtracts -g from the matrix diagonal (since dis/dvi = -g)
214 and adds i - g*(v+vext) to the
215 right hand side. In the presence of an extracellular mechanism, such electrode
216 mechanisms add the dis/dvi term to the extracellular equation diagonal and also add
217 is - dis/dvi*(viold) to the extracellular equation right hand side.
218 
219 This is where a trick comes in. Note that in the equations it is the
220 membrane currents that appear in both rows whereas the stimulus appears
221 in the internal row only. But the membrane mechanisms add membrane currents
222 only to the internal row and stimuli to both rows. Perhaps the additional
223 confusion was not worth the mechanism efficiency gain ( since there are many
224 more membrane (and synaptic) currents than stimulus currents) but at any rate
225 the matrix filling is ordered so that membrane mechanisms are done before
226 the axial currents. This means that only membrane and stimulus currents are present
227 on the internal diagonal and right hand side. However, only stimulus current
228 is present on the external diagonal and right hand side. The difference
229 of course is the membrane current elements by themselves and the extracellular
230 code fills the elements appropriately.
231 So, in the code below, the order for doing the extracellular setup is very
232 important with respect to filling the matrix and mechanisms must be first,
233 then the obsolete activsynapse, then the extracellular setup, then
234 linear models and the obsolete activstim and activclamp. I.e it is crucial
235 to realize that before extracellular has been called that the internal
236 current balance matrix rows are with respect to vm whereas (except for special
237 handling of stimulus terms by keeping separate track of those currents)
238 and after extracellular has been called the current balance matrix rows are
239 with respect to vi and vext.
240 Finally at the end it is possible to add the internal axial terms.

254 portion of those solvers, the reuse of the setup is wastful because
255 after setting up the matrix, we essentially calculate rhs - M*x
256 and then throw away M.
257

259 to re-implement the setup functions to separate the calculation of M
260 from that of b. Those pieces could then be used by all three methods as
261 needed. Also at this level of strategy we can properly take into
262 account cvode's and daspk's requirement of separating the dy/dt terms
263 from the y dependent terms. Note that the problem we are dealing with
264 is entirely confined to the current balance equations and little or
265 no change is required for the mechanism ode's.

269 performs can be split into two parts only if the jacabian calculation is done
270 first. This is because the first BREAKPOINT evaluation is for
271 v+.001 and the second is for v. We can't turn that around. The best that
272 can be done (apart from analytic calculation of g) is to make use
273 of the simultaneous calculation of g for the fixed step, store it for
274 use when the jacobian is evalulated and eventually

1481 static double diam_from_list(Section* sec, int inode, Prop* p, double rparent)
1482 /* p->param[0] is diam of inode in sec.*/
1483 /* rparent right half resistance of the parent segment*/
1484 {
1485  /* Basic algorithm assumes a set of monotonic points on which a
1486  function is defined. The extension is the piecewise continuous
1487  function. y(x) for 0<=x<=x[n] (extrapolate for x>x[n]).
1488  The problem is to form the integral of f(y(s)) over intervals
1489  s[i] <= s <= s[i+1]. Note the intervals don't have to be equal.
1490  This implementation is specific to a call to this function for
1491  each interval in order.
1492  */
1493  /* computes diam as average, area, and ri. Slightly weirder since
1494  interval spit in two to compute right and left half values of ri
1495  */
1496  /* fills NODEAREA and NODERINV and returns the right half resistance
1497  (MOhms) of the segment.
1498  */
1499  static int j;
1500  static double x1, y1, ds;
1501  int ihalf;
1502  double si, sip;
1503  double diam, delta, temp, ri, area, ra, rleft = 0.0;
1504  int npt, nspine;
1505 
1506  if (inode == 0) {
1507  j = 0;
1508  si = 0.;
1509  x1 = sec->pt3d[j].arc;
1510  y1 = fabs(sec->pt3d[j].d);
1511  ds = sec->pt3d[sec->npt3d - 1].arc / ((double) (sec->nnode - 1));
1512  }
1513  si = (double) inode * ds;
1514  npt = sec->npt3d;
1515  diam = 0.;
1516  area = 0.;
1517  nspine = 0;
1518  ra = nrn_ra(sec);
1519  for (ihalf = 0; ihalf < 2; ihalf++) {
1520  ri = 0.;
1521  sip = si + ds / 2.;
1522  for (;;) {
1523  int jp, jnext;
1524  double x2, y2, xj, xjp;
1525  jp = j + 1;
1526  xj = sec->pt3d[j].arc;
1527 #if NTS_SPINE
1528  if (sec->pt3d[j].d < 0 && xj >= si && xj < sip) {
1529  nspine++;
1530  }
1531 #endif
1532  xjp = sec->pt3d[jp].arc;
1533  if (xjp > sip || jp == npt - 1) {
1534  double frac;
1535  if (fabs(xjp - xj) < 1e-10) {
1536  frac = 1;
1537  } else {
1538  frac = (sip - xj) / (xjp - xj);
1539  }
1540  x2 = sip;
1541  y2 = (1. - frac) * fabs(sec->pt3d[j].d) + frac * fabs(sec->pt3d[jp].d);
1542  jnext = j;
1543  } else {
1544  x2 = xjp;
1545  y2 = fabs(sec->pt3d[jp].d);
1546  jnext = jp;
1547  }
1548 
1549  /* f += integrate(x1,y1, x2, y2) */
1550  delta = (x2 - x1);
1551  diam += (y2 + y1) * delta;
1552  if (delta < 1e-15) {
1553  delta = 1e-15;
1554  }
1555  if ((temp = y2 * y1 / delta) == 0) {
1556  temp = 1e-15;
1557  }
1558  ri += 1 / temp;
1559 #if 1 /*taper is very seldom taken into account*/
1560  temp = .5 * (y2 - y1);
1561  temp = sqrt(delta * delta + temp * temp);
1562 #else
1563  temp = delta;
1564 #endif
1565 
1566  area += (y2 + y1) * temp;
1567  /* end of integration */
1568 
1569  x1 = x2;
1570  y1 = y2;
1571  if (j == jnext) {
1572  break;
1573  }
1574  j = jnext;
1575  }
1576  if (ihalf == 0) {
1577  rleft = ri * ra / PI * (4. * .01); /*left seg resistance*/
1578  } else {
1579  ri = ri * ra / PI * (4. * .01); /* MegOhms */
1580  /* above is right half segment resistance */
1581  }
1582  si = sip;
1583  }
1584  /* answer for inode is here */
1585  NODERINV(sec->pnode[inode]) = 1. / (rparent + rleft);
1586  diam *= .5 / ds;
1587  if (fabs(diam - p->param(0)) > 1e-9 || diam < 1e-5) {
1588  p->param(0) = diam; /* microns */
1589  }
1590  sec->pnode[inode]->area() = area * .5 * PI; /* microns^2 */
1591 #if NTS_SPINE
1592  /* if last point has a spine then increment spine count for last node */
1593  if (inode == sec->nnode - 2 && sec->pt3d[npt - 1].d < 0.) {
1594  nspine += 1;
1595  }
1596  sec->pnode[inode]->area() = sec->pnode[inode]->area() + nspine * spinearea;
1597 #endif
1598  return ri;
1599 }

2040  *

2043  * computations performed while time-stepping.

2046  * It is guaranteed to remain "ready" until the returned tokens are destroyed.
2047  */
2048 static void nrn_sort_mech_data(
2049  neuron::container::Mechanism::storage::frozen_token_type& sorted_token,
2050  neuron::cache::Model& cache,
2051  neuron::container::Mechanism::storage& mech_data) {
2052  // Do the actual sorting here. For now the algorithm is just to ensure that
2053  // the mechanism instances are partitioned by NrnThread.
2054  auto const type = mech_data.type();

2056  // sorted

2061  [mech_data_size,

2064  auto field) {

2067  pdata_hack.resize(field + 1);
2068  }
2069  pdata_hack.at(field).reserve(mech_data_size);
2070  pdata_fields_to_cache.push_back(field);
2071  });
2072  std::size_t global_i{}, trivial_counter{};
2073  std::vector<std::size_t> mech_data_permutation(mech_data_size,
2074  std::numeric_limits<std::size_t>::max());

2077  // the Memb_list for this mechanism in this thread, this might be
2078  // null if there are no entries, or if it's an artificial cell type(?)
2079  auto* const ml = nt->_ml_list[type];
2080  if (ml) {
2081  // Tell the Memb_list what global offset its values start at
2082  ml->set_storage_offset(global_i);
2083  }
2084  // Record where in the global storage this NrnThread's instances of
2085  // the mechanism start
2086  cache.thread.at(nt->id).mechanism_offset.at(type) = global_i;
2087  // Count how many times we see this mechanism in this NrnThread
2088  auto nt_mech_count = 0;
2089  // Loop through the Nodes in this NrnThread
2090  for (auto i = 0; i < nt->end; ++i) {
2091  auto* const nd = nt->_v_node[i];
2092  // See if this Node has a mechanism of this type
2093  for (Prop* p = nd->prop; p; p = p->next) {
2094  if (p->_type != type) {
2095  continue;
2096  }
2097  // this condition comes from thread_memblist_setup(...)
2098  if (!memb_func[type].current && !memb_func[type].state &&
2099  !memb_func[type].has_initialize()) {
2100  continue;
2101  }
2102  // OK, p is an instance of the mechanism we're currently
2103  // considering.
2104  auto const current_global_row = p->id().current_row();
2105  trivial_counter += (current_global_row == global_i);
2106  mech_data_permutation.at(current_global_row) = global_i++;
2107  for (auto const field: pdata_fields_to_cache) {
2108  cache.mechanism.at(type).pdata_hack.at(field).push_back(p->dparam + field);
2109  }
2110  // Checks
2111  assert(ml->nodelist[nt_mech_count] == nd);
2112  assert(ml->nodeindices[nt_mech_count] == nd->v_node_index);
2113  ++nt_mech_count;
2114  }
2115  }
2116  assert(!ml || ml->nodecount == nt_mech_count);
2117  // Look for any artificial cells attached to this NrnThread
2118  if (nrn_is_artificial_[type]) {
2119  cTemplate* tmp = nrn_pnt_template_[type];

2121  ITERATE(q, tmp->olist) {
2122  Object* obj = OBJ(q);
2123  auto* pnt = static_cast<Point_process*>(obj->u.this_pointer);
2124  assert(pnt->prop->_type == type);
2125  if (nt == pnt->_vnt) {
2126  auto const current_global_row = pnt->prop->id().current_row();
2127  trivial_counter += (current_global_row == global_i);
2128  mech_data_permutation.at(current_global_row) = global_i++;
2129  for (auto const field: pdata_fields_to_cache) {
2130  cache.mechanism.at(type).pdata_hack.at(field).push_back(
2131  pnt->prop->dparam + field);
2132  }
2133  }
2134  }
2135  }
2136  }
2137  if (global_i != mech_data_size) {
2138  // This means that we did not "positively" find all the instances of
2139  // this mechanism by traversing the model structure. This can happen
2140  // if HOC (or probably Python) scripts create instances and then do
2141  // not attach them anywhere, or do not explicitly destroy
2142  // interpreter variables that are preventing reference counts from
2143  // reaching zero. In this case we can figure out which the missing
2144  // entries are and permute them to the end of the vector.
2145  auto missing_elements = mech_data_size - global_i;
2146  // There are `missing_elements` integers from the range [0 ..
2147  // mech_data_size-1] whose values in `mech_data_permutation` are
2148  // still std::numeric_limits<std::size_t>::max().
2149  for (auto global_row = 0ul; global_row < mech_data_size; ++global_row) {

2151  trivial_counter += (global_row == global_i);
2152  mech_data_permutation[global_row] = global_i++;
2153  --missing_elements;
2154  if (missing_elements == 0) {
2155  break;
2156  }
2157  }
2158  }
2159  if (global_i != mech_data_size) {

2286  */
2287 neuron::model_sorted_token nrn_ensure_model_data_are_sorted() {
2288  // Rather than a more complicated lower-level solution, just serialise all
2289  // calls to this method. The more complicated lower-level solution would
2290  // presumably involve a more fully-fledged std::shared_mutex-type model,
2291  // where the soa containers can be locked by many readers (clients that do
2292  // not do anything that invalidates pointers/caches) or a single writer
2293  // (who *is* authorised to perform those operations), with a deadlock
2294  // avoiding algorithm used to acquire those two types of lock for all the
2295  // different soa containers at once.
2296  static std::mutex s_mut{};
2297  std::unique_lock _{s_mut};
2298  // Two scenarii:
2299  // - model is already sorted, in which case we just assemble the return
2300  // value but don't mutate anything or do any real work
2301  // - something is not already sorted, and by extension the cache is not
2302  // valid.
2303  // In both cases, we want to start by acquiring tokens from all of the
2304  // data containers in the model. Once we have locked the whole model in
2305  // this way, we can trigger permuting the model (by loaning out the tokens
2306  // one by one) to mark it as sorted.
2307  auto& model = neuron::model();
2308  auto& node_data = model.node_data();
2309  // Get tokens for the whole model
2310  auto node_token = node_data.issue_frozen_token();
2311  auto already_sorted = node_data.is_sorted();
2312  // How big does an array have to be to be indexed by mechanism type?
2313  auto const mech_storage_size = model.mechanism_storage_size();
2314  std::vector<neuron::container::Mechanism::storage::frozen_token_type> mech_tokens{};
2315  mech_tokens.reserve(mech_storage_size);
2316  model.apply_to_mechanisms([&already_sorted, &mech_tokens](auto& mech_data) {
2317  mech_tokens.push_back(mech_data.issue_frozen_token());
2318  already_sorted = already_sorted && mech_data.is_sorted();
2319  });
2320  // Now the whole model is marked frozen/read-only, but it may or may not be
2321  // marked sorted (if it is, the cache should be valid, otherwise it should
2322  // not be).
2323  if (already_sorted) {
2324  // If everything was already sorted, the cache should already be valid.
2325  assert(neuron::cache::model);
2326  // There isn't any more work to be done, really.
2327  } else {
2328  // Some part of the model data is not already marked sorted. In this
2329  // case we expect that the cache is *not* valid, because whatever
2330  // caused something to not be sorted should also have invalidated the
2331  // cache.
2332  assert(!neuron::cache::model);
2333  // Build a new cache (*not* in situ, so it doesn't get invalidated
2334  // under our feet while we're in the middle of the job) and populate it
2335  // by calling the various methods that sort the model data.
2336  neuron::cache::Model cache{};
2337  cache.thread.resize(nrn_nthread);
2338  for (auto& thread_cache: cache.thread) {
2339  thread_cache.mechanism_offset.resize(mech_storage_size);
2340  }
2341  cache.mechanism.resize(mech_storage_size);
2342  // The cache is initialised enough to be populated by the various data
2343  // sorting algorithms. The small "problem" here is that all of the
2344  // model data structures are already marked as frozen via the tokens
2345  // that we acquired above. The way around this is to transfer those
2346  // tokens back to the relevant containers, so they can check that the
2347  // only active token is the one that has been provided back to them. In
2348  // other words, any token is a "read lock" but a non-const token that
2349  // refers to a container with token reference count of exactly one has
2350  // an elevated "write lock" status.
2351  nrn_sort_node_data(node_token, cache);
2352  assert(node_data.is_sorted());

2354  std::size_t n{}; // eww
2355  model.apply_to_mechanisms([&cache, &n, &mech_tokens](auto& mech_data) {
2356  // TODO do we need to pass `node_token` to `nrn_sort_mech_data`?
2357  nrn_sort_mech_data(mech_tokens[n], cache, mech_data);
2358  assert(mech_data.is_sorted());
2359  ++n;
2360  });
2361  // Now that all the mechanism data is sorted we can fill in pdata caches
2362  model.apply_to_mechanisms(
2363  [&cache](auto& mech_data) { nrn_fill_mech_data_caches(cache, mech_data); });
2364  // Move our working cache into the global storage.
2365  neuron::cache::model = std::move(cache);
2366  }

2368  neuron::model_sorted_token ret{*neuron::cache::model, std::move(node_token)};

2370  return ret;
